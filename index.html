转型关键期，
为什么现在需要 AI 提效

迪尚华盛正由 OEM 向 ODM 转型，业务量与文档量持续增加，人工处理的边界正在被打破。

现状背景

  • 业务量与客户对接文档量持续增加，跨部门协同愈发频繁
  • 日常工作高度依赖邮件往来和 WPS / Excel 表格整理
  • 涉及订单确认、面辅料准备、报价核价、生产跟单、包装发货、结汇对账等十余个环节
  • 信息量大、节点多，容易因人工疏漏影响交期与品质

企划目标

  • 把业务员 / 跟单员从重复性信息整理、表格制作、翻译起草中解放出来
  • 把更多精力用于客户沟通、异常判断与经营决策
  • 降低因信息遗漏导致的交期、品质风险

六 · 进阶方案

接入 API：从"手动搬运"到"自动运行"

现在的用法仍需要员工每一步手动操作，接入 API 后 AI 可在后台自动运行。

现状

目前用法本质是"人工打开网页对话框 → 手动复制粘贴内容 → 等待AI回复 → 再手动复制回表格/邮件"，AI 仍需要员工每一步手动操作。

接入 API 后（企业购买调用额度，接入 WPS / 邮箱 / 企业微信等系统），AI 可在后台自动运行，员工只需点一下按钮、上传文件，甚至无需操作，最后只做结果确认。

实施前提

  • 需公司安排懂技术的人员（IT同事或外部服务商）把 API 接到具体的小工具里，属于一次性开发投入
  • 开发完成后，员工使用方式与点开网页按钮相当简单，不需要写代码或接触 API Key
  • 建议先挑 1-2 个高频场景做试点小工具，验证好用后再逐步扩展

同样是这 9 个场景，现在靠人工打开对话框操作；接入 API 之后，多数环节可以自动触发，人只需要最后确认结果。

DeepSeek API 与豆包（火山方舟）API 该怎么选

对比维度DeepSeek API豆包 / 火山方舟 API

建议：涉及图片场景（样衣照片处理、纸质单据OCR）优先豆包/火山方舟；知识库问答优先豆包；纯文本强逻辑核算（报价核价、多表交叉校验）优先 DeepSeek；其余文本类场景（邮件摘要、翻译、交期预警、生产周报）两者能力接近，可先各选一个小范围试用后再定，不必只选一个全部替换。

实施路径

试点 → 推广 → 固化，三阶段稳步推进，不做一次性大范围切换。
